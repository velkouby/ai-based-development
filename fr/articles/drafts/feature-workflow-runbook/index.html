    AppForge Feature Workflow Runbook¶

    Ce document decrit le workflow AppForge pour livrer une feature depuis sa creation jusqu'a sa fermeture. Il sert a expliquer ce que fait chaque commande, quels fichiers sont produits, et quelles sous-etapes verifier pendant un run.

    Note de syntaxe: la commande correcte est appforge feature, pas appforge fearture. Dans la CLI actuelle, --yolo n'est pas supporte par appforge feature plan; il s'utilise sur appforge feature run, appforge feature resume, ou appforge feature guided --confirm.

    Sequence recommandee¶

    appforge feature create feature-name
# edit .appforge/features/feature-name/brief.md
appforge feature plan feature-name
appforge feature run feature-name --live --yolo
appforge check quality --profile default
appforge repair quality
appforge check quality --profile default
appforge check feature feature-name
appforge feature finalize feature-name

    Pour un run plus controle, remplacer appforge feature run feature-name --live --yolo par des runs de taches:

    appforge task run feature-name task-001
appforge task run feature-name task-002
appforge feature status feature-name

    1. Creer la feature¶

    Commande:

    appforge feature create feature-name

    Ce que fait AppForge:

    • verifie que le dossier courant est un projet AppForge genere;
    • valide le slug feature-name;
    • cree ou checkout la branche Git appforge/feature-name si le projet est un depot Git;
    • cree le dossier .appforge/features/feature-name/;
    • ecrit un brief initial dans .appforge/features/feature-name/brief.md;
    • ecrit l'etat machine dans .appforge/features/feature-name/feature.json.

    Fichiers importants apres creation:

    .appforge/features/feature-name/
|-- brief.md
|-- feature.json

    Etat attendu:

    • feature.json contient status: draft;
    • plan.md et tasks.json n'existent pas encore;
    • le brief contient une structure de depart: objectif, scope, criteres d'acceptation, notes.

    2. Ecrire le brief¶

    Action humaine:

    $EDITOR .appforge/features/feature-name/brief.md

    Le brief est le contrat produit de depart. Il doit contenir assez de contexte pour que l'agent puisse planifier sans inventer les intentions produit.

    Sections utiles:

    • Objective: resultat utilisateur visible a construire;
    • Scope: ce qui est inclus et exclu;
    • Acceptance Criteria: criteres mesurables qui diront si la feature est terminee;
    • Notes: contraintes, liens Figma, decisions connues, donnees de test, risques.

    Bonne pratique:

    • utiliser des criteres d'acceptation verifiables;
    • nommer les surfaces concernees: backend, frontend, Android;
    • indiquer les chemins ou domaines metier attendus quand ils sont connus;
    • mettre explicitement les choix hors scope pour eviter que l'agent elargisse la feature.

    3. Planifier la feature¶

    Commande:

    appforge feature plan feature-name

    Options utiles:

    appforge feature plan feature-name --runner claude
appforge feature plan feature-name --mock
appforge feature plan feature-name --no-interactive
appforge feature plan feature-name --debug-runtime

    Ce que fait AppForge:

    • resout le slug depuis l'argument, la branche appforge/<slug>, ou l'unique feature presente;
    • recharge le brief;
    • genere ou recharge le contrat AppForge du projet;
    • collecte les instructions agent, les docs agent-docs/, les sources design detectees et un apercu des fichiers du repo;
    • ajoute les reponses utilisateur deja stockees dans answers.json si des questions avaient ete posees;
    • appelle le runner agent pour produire un plan structure;
    • exige un plan_markdown, une liste de tasks, des commandes de validation, des questions ouvertes et des warnings;
    • refuse le plan si le markdown ne contient pas de section Write Boundary Matrix;
    • normalise les taches en ids stables task-001, task-002, etc.;
    • valide les chemins autorises, read-only et interdits contre le contrat AppForge;
    • ecrit .appforge/features/feature-name/plan.md;
    • ecrit .appforge/features/feature-name/tasks.json;
    • met a jour feature.json.

    Fichiers importants apres planification:

    .appforge/features/feature-name/
|-- brief.md
|-- feature.json
|-- plan.md
|-- tasks.json
|-- answers.json          # cree ou complete si des questions humaines sont necessaires

    Etat attendu:

    • feature.json passe a planned si aucune question n'est ouverte;
    • feature.json passe a needs_user_input si l'agent demande des decisions;
    • tasks.json contient les taches, leurs dependances, leurs chemins autorises et leurs validations;
    • chaque tache garde son historique d'essais dans attempts.

    Si AppForge pose des questions:

    appforge feature status feature-name
$EDITOR .appforge/features/feature-name/answers.json
appforge feature plan feature-name

    En mode interactif, la CLI peut demander les reponses directement puis relancer la planification une fois.

    4. Executer les taches¶

    Commande autonome recommandee:

    appforge feature run feature-name --live --yolo

    Sans --yolo:

    appforge feature run feature-name --live

    Ce que fait AppForge:

    • charge feature.json et tasks.json;
    • bloque si une tache precedente est blocked, failed, needs_user_input ou needs_retry;
    • selectionne les taches pending dont les dependances sont satisfaites;
    • construit un paquet d'execution avec brief, plan, contrat, matrice des droits d'ecriture, reponses utilisateur et contexte Git;
    • lance le runner agent sur les taches eligibles;
    • attend un resultat structure avec status, resume, fichiers modifies, validations, questions, bloqueurs et warnings;
    • verifie que les fichiers modifies respectent allowed_paths, read_only_paths et forbidden_paths;
    • lance les boundary checks;
    • lance les commandes de validation declarees par les taches;
    • lance aussi appforge check quality --profile default comme evidence qualite quand les gates precedentes passent;
    • ecrit une evidence JSON sous .appforge/features/feature-name/evidence/;
    • met a jour tasks.json et feature.json.

    Avec --live, les evenements du workflow sont affiches pendant l'execution. Avec --yolo, AppForge peut accepter certaines recommandations basse criticite ou tenter des recoveries automatiques si la politique du projet l'autorise. Les decisions sensibles restent humaines.

    Fichiers produits pendant le run:

    .appforge/features/feature-name/
|-- events.jsonl
|-- evidence/
|   |-- feature-run-attempt-001.json
|-- interventions/
|-- recovery/
|-- sessions/

    Etats possibles:

    • in_progress: certaines taches sont terminees, d'autres restent a faire;
    • completed: toutes les taches sont done;
    • needs_user_input: une decision humaine est necessaire;
    • blocked ou failed: un bloqueur, une violation de policy, une validation ou un runner a echoue.

    Pour reprendre apres un arret:

    appforge feature status feature-name
appforge feature interventions feature-name
appforge feature resume feature-name --live

    Pour reessayer une tache specifique:

    appforge task retry feature-name task-001

    5. Controler la qualite¶

    Commande:

    appforge check quality --profile default

    Ce que fait AppForge:

    • detecte les surfaces activees du projet: backend, frontend, Android;
    • charge appforge.quality.toml si present, sinon les checks par defaut;
    • selectionne les checks du profil default;
    • execute les commandes avec leur dossier de travail et timeout;
    • parse les sorties des outils supportes;
    • normalise les diagnostics;
    • groupe les diagnostics en repair blocks;
    • ecrit un rapport JSON dans .appforge/quality/reports/;
    • retourne un code non-zero si un check bloquant echoue.

    Exemples de checks par defaut:

    • backend Python: ruff check, ruff format --check, pyflakes, python backend/manage.py check;
    • frontend: eslint, tsc --noEmit, prettier --check;
    • Android: ./gradlew lint, ./gradlew spotlessCheck.

    Sortie attendue:

    [OK] quality report: <report-id>
Report: .appforge/quality/reports/<report-id>.json
Checks: <n> passed, <n> failed, <n> environment issue(s).
Diagnostics: <n> unresolved / <n> total.

    Si le rapport est [FAIL], ouvrir le JSON pour voir:

    • checks: resultat brut par outil;
    • diagnostics: erreurs normalisees avec fichier, ligne, code et message;
    • repair_blocks: groupes que appforge repair quality essaiera de corriger.

    6. Reparer la qualite¶

    Commande:

    appforge repair quality

    Variantes:

    appforge repair quality <report-id>
appforge repair quality --report .appforge/quality/reports/<report-id>.json
appforge repair quality --mechanical-only
appforge repair quality --runner claude

    Ce que fait AppForge:

    • charge le dernier rapport qualite si aucun id ou path n'est donne;
    • regroupe les diagnostics non resolus par repair block;
    • ignore les problemes d'environnement ou de configuration qui ne sont pas des erreurs de code;
    • tente d'abord les reparations mecaniques disponibles;
    • par exemple ruff --fix, ruff format, eslint --fix, prettier --write, spotlessApply;
    • relance les commandes de verification associees au block;
    • marque les diagnostics fixed, failed, skipped ou not_fixable;
    • si un runner agent est configure et que la reparation mecanique n'est pas suffisante, construit un prompt limite aux diagnostics de code eligibles;
    • ecrit les traces de reparation dans le rapport qualite.

    Apres reparation, toujours relancer:

    appforge check quality --profile default

    La reparation peut modifier du code. Il faut donc verifier aussi le statut Git:

    git status --short

    7. Verifier la feature¶

    Commande:

    appforge check feature feature-name

    Ce que fait AppForge:

    • verifie la coherence du workflow;
    • verifie que plan.md et tasks.json existent pour une feature planifiee;
    • verifie l'unicite des ids de taches;
    • verifie les champs obligatoires des taches;
    • verifie les dependances;
    • verifie les politiques de chemins;
    • verifie les evidences pour les taches done;
    • execute aussi les boundary checks du projet;
    • signale les duplications UI ou imports directs interdits selon les politiques AppForge.

    Cette etape est importante avant finalize, car feature finalize exige que les checks feature passent.

    8. Finaliser la feature¶

    Commande:

    appforge feature finalize feature-name

    Ce que fait AppForge:

    • relit tasks.json;
    • refuse la fermeture si une tache n'est pas done;
    • lance les checks feature;
    • refuse la fermeture si un check feature echoue;
    • genere ou regenere une review finale;
    • ecrit .appforge/features/feature-name/reviews/final-review.md;
    • met feature.json a status: completed;
    • ajoute finalized_at;
    • enregistre last_action.name: feature_finalize.

    Fichiers de cloture:

    .appforge/features/feature-name/
|-- feature.json
|-- reviews/
|   |-- final-review.md

    La review finale resume:

    • le brief;
    • le plan execute;
    • les taches et leur statut;
    • les fichiers modifies;
    • les evidences de run;
    • les validations;
    • les risques et questions restantes.

    Exemple complet commentable¶

    # 1. Creation de la feature et de sa branche appforge/feature-name.
appforge feature create feature-name

# 2. Description produit humaine.
$EDITOR .appforge/features/feature-name/brief.md

# 3. Transformation du brief en plan et taches.
appforge feature plan feature-name

# 4. Execution autonome des taches eligibles.
appforge feature run feature-name --live --yolo

# 5. Inspection de l'etat courant.
appforge feature status feature-name

# 6. Qualite globale du projet.
appforge check quality --profile default

# 7. Reparation si le rapport qualite echoue.
appforge repair quality
appforge check quality --profile default

# 8. Controle des gates specifiques a la feature.
appforge check feature feature-name

# 9. Fermeture durable de la feature.
appforge feature finalize feature-name

    Points de controle rapides¶

    • Apres create: brief.md et feature.json existent.
    • Apres edition du brief: l'objectif, le scope et les criteres d'acceptation sont clairs.
    • Apres plan: plan.md contient Write Boundary Matrix et tasks.json contient des task-001, task-002, etc.
    • Apres run: chaque tache terminee a une evidence sous evidence/.
    • Apres check quality: le dernier rapport sous .appforge/quality/reports/ est vert.
    • Apres repair quality: relancer check quality, car repair modifie le code puis annote le rapport.
    • Avant finalize: appforge check feature feature-name passe.
    • Apres finalize: feature.json contient status: completed et finalized_at.
